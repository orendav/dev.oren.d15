1. Overview

Every homepage image (~158 references) is served via Imgix from a Cloudflare R2 bucket. The R2 bucket is wbt; Imgix host is https://wbt01.imgix.net/. There's no build step — URLs are static strings inline in HTML src/srcset attributes. The first request to a given URL pulls from R2; every request after that is served from Imgix's CDN edge.

Imgix host
https://wbt01.imgix.net/
R2 bucket
wbt (mapped as Imgix source root)
Total imgix URLs in v5
~114 unique base URLs × 4–5 srcset variants = ~600 cached variants
Default response format
AVIF (Chrome/Edge) → WebP (Firefox/older Safari) → JPEG (fallback)

2. URL shape

Three R2 prefixes are in use — pick the right one for the asset:

PrefixWhat's thereExample
test01/09 Activewear/<SKU>/ Source-library Biz Collection talent + product photos for most SKUs test01/09 Activewear/SW225M/SW225M_Talent_Navy_White_01_R.jpg
test01/<SKU>/ flat Two SKUs uploaded later (J3150, J408M) live flat under test01/ instead of nested under 09 Activewear. Historical accident — keep aware when adding URLs for those two SKUs. test01/J408M/J408M_Talent_Black_Red_02_R.jpg
test01/bundles/ Project-generated bundle / pack composites (AI gpt-image PNGs + PIL composites). Not in source library — uploaded explicitly. test01/bundles/ai-bundle-print-tee-fb.png

Spaces in the path must be URL-encoded as %20. The full URL pattern:

https://wbt01.imgix.net/test01/09%20Activewear/{SKU}/{filename}.jpg?{params}

Mandatory params on every URL

ParamValueWhat it does
autoformat,compressImgix sniffs the request's Accept header and serves AVIF / WebP / JPEG accordingly, plus applies smart quality reduction. Single biggest win.
winteger pxOutput width. Always paired with srcset variants — see §3.

Conditional params

ParamValueUsed when
fitcropRequired whenever ar or both w+h are passed. Tells imgix to crop to the requested shape.
cropfaces / entropy / focalpointHow to choose what to keep when cropping. See §4.
ar8:3, 4:3, etc.Force output aspect ratio. Combined with fit=crop&crop=faces this is what makes wide banners not clip people's heads. See §5.
fp-x, fp-y0–1 floatsManual focal point. Only used as a fallback when face detection misses (none currently in use).

3. srcset / sizes / DPR — three patterns

The browser picks the right variant from srcset based on viewport, CSS sizing, and device pixel ratio. We use one of three patterns depending on whether the image's rendered size is fluid or fixed.

Two crop modes are in play, depending on what's in the image:
  • People (talent shots) → fit=crop&crop=faces&ar=W:H. Imgix face-aware-crops to the container's aspect, then browser object-fit: cover is a no-op. Faces stay in frame.
  • Products (flat studio photos) → fit=fill&fill=solid&fill-color=fff&ar=W:H. Imgix returns the image at the container's aspect with white padding on whatever side wouldn't fit. Full garment stays visible — collar to hem — instead of getting top/bottom cropped through the chest.

Pattern A — Width-described srcset + sizes fluid

Used for any image whose CSS width changes with viewport (banners, product cards, hero, featured banner, journal cards). The browser reads sizes to learn how wide the image will render, then picks the smallest srcset variant ≥ that width × DPR.

<img src="images/talent/jacket-model-1.jpg"
     srcset="https://wbt01.imgix.net/.../TP409M_Talent_Black_01_R.jpg?auto=format,compress&fit=crop&crop=faces&ar=8:3&w=360 360w,
             https://wbt01.imgix.net/.../TP409M_Talent_Black_01_R.jpg?auto=format,compress&fit=crop&crop=faces&ar=8:3&w=540 540w,
             https://wbt01.imgix.net/.../TP409M_Talent_Black_01_R.jpg?auto=format,compress&fit=crop&crop=faces&ar=8:3&w=720 720w,
             https://wbt01.imgix.net/.../TP409M_Talent_Black_01_R.jpg?auto=format,compress&fit=crop&crop=faces&ar=8:3&w=960 960w,
             https://wbt01.imgix.net/.../TP409M_Talent_Black_01_R.jpg?auto=format,compress&fit=crop&crop=faces&ar=8:3&w=1280 1280w"
     sizes="(min-width: 992px) 60vw, 100vw"
     alt="" loading="lazy" decoding="async">

Pattern B — DPR srcset (1x / 2x) fixed

Used for tiles whose CSS pixel size is constant — mega-menu thumbnails (~80px), bulk-order grid swatches, gender tiles. sizes attribute is omitted.

<img src="images/products/quarter-zip-navy-white.jpg"
     srcset="https://wbt01.imgix.net/.../SW225M_Product_Navy_White_01.jpg?auto=format,compress&fit=crop&crop=entropy&w=200 1x,
             https://wbt01.imgix.net/.../SW225M_Product_Navy_White_01.jpg?auto=format,compress&fit=crop&crop=entropy&w=400 2x"
     alt="" loading="lazy">

Pattern C — Single URL fixed

Used where srcset can't go: <video poster=...>, data-hover-images (loaded once on hover), data-image-alt on clearance cards (variant preview swap).

<video poster="https://wbt01.imgix.net/.../Talent_Group_Flex_01_R.jpg?auto=format,compress&fit=crop&crop=faces&ar=1:1&w=1024" ...>

4. Crop modes

CropWhen to useSetup
faces Any image with people. Imgix's face detector finds faces and biases the crop to keep them in frame. Works on single models, group shots, off-centre subjects. Auto. Just pass crop=faces.
fit=fill Now used for all flat product photos and bundle/pack composites. Imgix pads with white instead of cropping — full garment from collar to hem stays visible. Replaced the earlier crop=entropy approach which still cropped through the body when the container's aspect didn't match the source's 2:3 portrait. fit=fill&fill=solid&fill-color=fff&ar=W:H
entropy (legacy) Imgix picks the high-contrast region. Used previously for product photos — replaced because cropping through the chest of a long garment looked bad. crop=entropy — not used in v5 anymore.
focalpoint Manual override when faces / entropy both miss. Pass coordinates as 0..1 floats. Not currently used; reserve as a fallback. crop=focalpoint&fp-x=0.5&fp-y=0.3

5. ar= aspect ratio — the gotcha that almost shipped

If you only specify w=: Imgix returns the image at the source aspect (most lifestyle shots are 2:3 portrait). The browser then does object-fit: cover centred-vertical to fit a wide banner — clipping the model's head off. Face detection only positions inside imgix's output; it doesn't help once object-fit takes over.
The fix: Pass ar=W:H matching the rendered banner's aspect. Imgix returns the image already cropped to that aspect, with crop=faces keeping faces in the visible band. Browser object-fit: cover becomes a no-op.
SectionContainer aspect (rendered)ar=
Hero video poster~525×560 desktop (square-ish)1:1
Banner rotator slides~723×272 desktop (very wide)8:3
Tertiary / Quaternary cards~361×272 desktop4:3
Featured banner image50% × min-height 4003:2
Mega-menu Gender tiles~397×200 (about 2:1)2:1
Mega-menu Category / Industry thumbs~321×140 (about 2.3:1) — set by grid-auto-rows: 140px in CSS2:1
Shop-by-Category homepage tiles4:5 desktop, 16:10 mobile4:5
Product cards1:1 (square)none — product photos already fill the frame, crop=entropy handles centring

Rule of thumb: any container that's a different aspect from the source's 2:3 portrait needs an ar= matching the container — otherwise the browser's object-fit: cover trims through the wrong axis and clips heads.

14. Hover-rotation (data-hover-images)

Bundle and pack cards swap their composite image for individual product photos as the mouse moves across the card. The list of swap-target URLs lives in a data-hover-images attribute. Important: the separator is |, not , — because imgix URLs themselves contain auto=format,compress, splitting on comma would fragment URLs.

<a class="bundle-card"
   data-hover-images="https://wbt01.imgix.net/.../A.jpg?auto=format,compress&fit=crop&crop=entropy&w=480 |
                      https://wbt01.imgix.net/.../B.jpg?auto=format,compress&fit=crop&crop=entropy&w=480 |
                      https://wbt01.imgix.net/.../C.jpg?auto=format,compress&fit=crop&crop=entropy&w=480">

The split is in js/main.js:

const hoverList = card.dataset.hoverImages.split('|').map(s => s.trim()).filter(Boolean);

16. Operational notes

Adding a new image

  1. Upload the source file to R2 under one of the three prefixes (§2). Test with curl -I https://wbt01.imgix.net/{key}?w=200 — should return 200.
  2. Pick the right pattern (§3) based on whether the rendered size is fluid or fixed.
  3. Pick the right crop (§4) — faces for people, entropy for products.
  4. Pick ar= (§5) only if the rendered container has a wide-or-tall aspect that doesn't match the source.
  5. Always include auto=format,compress.
  6. Keep the local file as src= fallback.

The migration script

designs/03-hyper.v5/scripts/imgix_migrate.py is the re-runnable preprocessor that does the bulk of the rewriting. It detects the section by line range and applies the right pattern. Idempotent for <img> tags (skips ones with srcset already).

Imgix billing model

Each unique URL is one origin pull (then cached at the CDN edge indefinitely). Width-ladder + section count = ~600 unique variants for v5. After first hit each is served from edge cache. Free tier is 1000 origin images / 100GB bandwidth per month — we're nowhere near that.

When face detection misses

Imgix's face detector occasionally misses on group shots, profile shots, or low-contrast lighting. Symptom: face is off-screen even with crop=faces&ar=…. Fix: switch that one URL to crop=focalpoint&fp-x=0.5&fp-y=0.3 (upper-third). Currently no images need this, but reserve the pattern.
